- **v0.7.9 (proposed) â€” Shared Context Layer implementation plan + design clarifications.**

  **New doc:** [`26-shared-context-layer-implementation.md`](./26-shared-context-layer-implementation.md)

  skill source bodies live in Git/UC Volumes while Lakebase stores only

- **v0.7.9 (proposed) â€” Context Intelligence Layer added.**

  **Decision:** Context Intelligence is the differentiating layer above Shared
  Context Graph memory. It turns Hot Context, Warm Context, and Cold Context
  into scoped, provenance-backed Prompt Context Packs for agents.

  **Design lock:** agents should not receive raw graph dumps or oversized
  prompts by default. A Selective Context Router chooses context by task,
  trust level, recency, dependency path, semantic match, and budget. Context
  selection decisions are stored as Reasoning Memory so the UI can explain
  why an agent knew something.

  **Implementation cut-line:** the first concrete build is a narrow Context
  Pack Builder for Skill Builds with Agents. The broader Context Intelligence
  Service remains the north-star layer.

- **v0.7.9 (proposed) â€” Shared Context Graph storage and skill location lock.**

  **Decision:** Skill Builds with Agents use a real Shared Context Graph, not
  prompt-only memory, manual status rows, or a table of ids. The graph has three
  memory layers: Long-Term Memory for reusable enterprise knowledge, Shared
  Working Memory for live collaboration, and Reasoning Memory for decision
  traces.

  **Storage lock:** Lakebase is the live operational source of truth for context
  events, current nodes/edges, Skill Requests, Skill Build Runs, and fast UI
  reads. UC Volumes hold large draft/build artifacts. Vector Search indexes
  selected semantic summaries. UC Delta is optional archive/export/offline
  analytics, not the live graph store.

  **Skill location lock:** approved BrickVision skills live under `skills/`;
  approved partner skills live under `skill-packs/<partner>/`; UC Volumes stage
  draft artifacts during build/review; Lakebase stores lifecycle metadata and
  Skill Knowledge Graph / Shared Context Graph state.

- **v0.7.9 (proposed) â€” Skill Library and three-graph architecture vocabulary lock.**

  **Decision:** The top-level skill product surface is **Skill Library**, not
  Skill Builder, Skill Catalog, Skill Workshop, Skill Gap, or Capability
  Backlog. Skill Library contains Available Skills, Skill Requests, Skill
  Builds, Draft Skills, and Partner Skills. "Build" is a lifecycle inside the
  Skill Library, not the product name.

  **Architecture lock:** BrickVision separates three knowledge substrates:
  Databricks Capability Graph = what is possible on Databricks; Skill Knowledge
  Graph = what BrickVision and partner skills can do; Workspace Context Graph =
  what exists in the active workspace. Agents join all three at planning time,
  but the stores remain separate.

  **PRD updates:** `docs/24-agent-operating-model.md` now uses Skill Library,
  Skill Request, Skill Build, Draft Skill, Partner Skill, Skill Knowledge Graph,
  and Workspace Context Graph terminology. The old Skill Gap protocol is renamed
