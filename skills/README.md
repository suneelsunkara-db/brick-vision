# BrickVision Skills

Each subfolder is one Skill. Every Skill has a `SKILL.yaml` that fully describes
what it does, when to use it, what it requires, what it emits into the KG, what
its evals look like, and what constitutional rules govern it.

The `SKILL.yaml` shape is the harness's source of truth â€” bad YAML = build
fails, runtime never sees it.

### ML architecture principle (v0.7.9)

The ML skill chain follows the same pattern as SQL and PySpark: each step is a
contracted skill with typed inputs/outputs. No monolithic training code. The
`lakeflow.jobs-run-submit` skill is the only execution boundary â€” it submits
code generated by skill plans, not handwritten. See
`docs/24-agent-operating-model.md` Â§24.6.A.

## SKILL.yaml shape (canonical)

Every skill must declare:

- **identity**: `id`, `version`, `owner`, `signing_key_id`
- **discoverability**: `title`, `description`, `when_to_use[]`, `load_signal.{embed, triggers[]}`
- **dependencies**: `requires.{uc_permissions[], system_schemas[], tools[], skills[], fms_endpoint, budget}`
- **contract**: `inputs[]`, `outputs[]` (typed)
- **execution detail**: `queries`/`operations` (mechanical skills) or implicit (LLM skills)
- **eval**: `eval.{suite, required_pass_rate, golden_dataset, scorers[]}`
- **safety**: `constitutional[]` rule references + `execution.human_in_the_loop` policy
- **graph link**: `exemplar_of: meta:<m>/ext:<e>` (mandatory â€” enforced by scorer)

A skill that fails to ship valid `SKILL.yaml` cannot register. A skill that
fails its declared eval pass rate cannot promote to `production` in MLflow Model
Registry.
